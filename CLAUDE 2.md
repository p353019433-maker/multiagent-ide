# 项目宪章 · Project Charter

> 这是本仓库的**最高优先级文档**。所有参与开发的 agent / 贡献者都应先读它，并以它为统一基准。
> 当不同 agent 的判断（尤其是安全口径、是否加确认、是否放开自动化）发生冲突时，**以本文件为准**；
> 本文件由项目负责人 + 主导 agent（Claude）维护，**不要在未对齐本文件的情况下擅自反向加码**。
>
> 一句话定位：**这是一个给作者本人用的、本地优先的 AI 代码编辑器**，目标是好用、自动、聪明，
> 不是一个"绝对安全"的多租户产品。安全性按**本机单用户环境**裁剪，而不是按防范恶意攻击者来裁剪。

---

## 1. 最终形态（我们要做成什么）

一个对标 Cursor / Codex / Antigravity 的桌面 AI IDE，但**面向个人本地使用**：

- 强大的 **Agent 编程**：能自主读写代码、跑命令、检索代码库、并行编排多个子任务，并**自我验证**结果。
- **多供应商 LLM**，OpenAI 兼容 + Anthropic；作者主力用国产模型（DeepSeek V4-pro / V4-flash 等），
  这类模型的能力/接口（FIM、embedding、长上下文）要优先照顾。
- 体验向主流 IDE 看齐：内联补全、语义/混合检索、代码导航、diff/检查点、Artifacts 等。
- **本地优先**：配置、会话、索引、Agent 工作区默认都留在本机，不依赖云端账户/服务。

## 2. 核心原则与优先级

当需要取舍时，按这个顺序：

1. **实用性 Practicality** — 真能解决作者的实际编程问题，开箱即用，不绕路。
2. **自动化 Automation** — 能自动完成的就不要让人点；默认朝"自动跑通"设计，而不是默认保守。
3. **智能化 Intelligence** — 充分利用模型能力（自我验证、自我修复、上下文检索、编排）。
4. **性能 Performance** — 大项目不卡死（重活进 worker、增量、缓存）。
5. **整洁 / 可维护** — 测试、类型、结构债。重要但排在体验之后。

> 「绝对安全」「企业级合规」「多用户隔离」**不在**优先级列表里——见第 3 节。

## 3. 安全口径（统一标准 · 重点）

这是各 agent 最容易分歧的地方，这里一次性定死。

### 3.1 威胁模型（我们防什么）

**单用户 / 本地 / 作者自己的机器。** 要防的只有两类：

- **模型/Agent 手滑造成的不可逆或对外破坏**：删库、`git push --force`、误开 PR/Release、格式化磁盘、把工作区外的东西改坏。
- **程序 bug 写坏用户数据/配置**。

**明确不在威胁模型内**（不要为它们牺牲实用性/自动化）：

- 恶意攻击者、多租户/SaaS 隔离、不可信第三方代码沙箱、供应链投毒、不可信前端页面。
- 本机文件、本机命令、用户自己配置的 provider 默认**可信**。

### 3.2 保留的护栏（这些是"防手滑 / 防 bug"，不是"防攻击者"，所以留着）

- 危险 shell 命令拦截：`rm -rf`、`dd of=`、`mkfs`、fork bomb、`git push --force`、`git reset --hard`、`git clean -f` 等（见 `src/shared/command-policy.ts`）。
- **对外 / 不可逆**操作默认要人确认或留给手动：`git push`、所有 `github_*` 写操作、PR/Release、分支合并到基线。
- 工作区路径围栏：Agent 不在"已授权工作区"之外乱写（`assertAllowedPath` 等）。把它当成"防手滑越界"，不是"防越权攻击"。
- API Key 用 Electron `safeStorage` 加密存储。

### 3.3 不要做的过度加固（除非作者明确要求）

- ❌ 不要把它做成强隔离沙箱 / seccomp / 容器化执行 / webview 强隔离到影响功能。
- ❌ 不要按多用户/SaaS/不可信前端的假设收紧 IPC、store、命令，到**影响正常功能或自动化流程**。
- ❌ 不要为"理论攻击面"增加确认弹窗、打断自动化、或默认拒绝。
- ❌ 不要默认不信任本机环境（文件、命令、provider）。

### 3.4 默认取舍

- 拿不准时，**选"能自动跑通"的那条**；只把**不可逆 / 对外**的留给人确认。
- 新功能**默认开自动化**，而不是默认保守。
- 后台/无人值守的 Agent：允许读、写工作区、跑安全命令、做本地 git；只挡对外写、分支合并、高危命令（见 `src/renderer/agent/headlessAgent.ts` 的策略）。

## 4. Agent 自治与自动化期望

- 编排（orchestrate）必须**名副其实**：子 Agent 要真有工具、能在隔离 worktree 里干活、并尽量**自我验证**（改完跑 lint/tsc，有错自己修一轮）。
- 优先做"少打扰人"的设计：能自动检测/重试/自愈的，不要弹窗问。
- 失败要可观测、可恢复（检查点、清晰的错误、可重试），但不要因为怕失败就不自动化。

## 5. 工程不变量（改动必须守住）

- **架构**：Electron 三进程（main / preload / renderer），IPC 经 `ipcMain.handle`；renderer 不直接碰 Node。保留 `contextIsolation`、preload 白名单。
- **质量门**：提交前 `npm test` 全绿、`tsc -p tsconfig.main.json` 与 `tsc -p tsconfig.json` 均 0 错误。新增逻辑配单测。
- **不泄密**：不要把 API Key、模型内部 ID、密钥写进仓库、提交信息、代码注释或日志。
- **依赖克制**：能不加重依赖就不加；打包相关（native / wasm / asar）改动要能在打包后验证，验证不了就别贸然合（参考 tree-sitter 的取舍记录）。
- **分支/PR**：在指定 feature 分支开发，提交信息清晰；未经作者同意不开/不合 PR。

## 6. 范围

**做**：Agent 编程、多供应商、检索（符号/AST/向量/BM25/混合/重排）、内联补全/FIM、代码导航、diff/检查点、Artifacts、多会话/worktree 编排、性能（worker 索引）。

**暂缓 / 不追**（作者已明确）：

- **MCP**：工作量大，暂缓。
- **强隔离沙箱**：与本定位冲突，不做。
- **VS Code 插件生态 / extension host**：不追；要等价能力时优先走 **LSP / DAP** 标准协议，而不是重写成插件宿主。

## 7. 怎么改这份宪章

- 这是**活文档**。要调整方向（比如想更激进地放开自动 push / 开 PR，或反过来收紧某处），**先改这里并与作者对齐**，再改代码。
- 具体的大决策（为什么这么做、放弃了什么）记录在提交信息里；本文件只保留**长期生效的原则**。
