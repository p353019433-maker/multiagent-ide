# 全维度全量代码审查设计

**日期**：2026-06-15
**项目**：AI Code IDE
**状态**：已批准，待执行
**作者**：ZCode（brainstorming 流程产出）

## 1. 背景与目标

`AI Code IDE` 是一个基于 Electron + React + TypeScript 的桌面 AI 代码编辑器，包含主进程 services、40+ Agent 工具、6 个 React context、多个组件层，以及 embedding 语义检索、FIM 内联补全、检查点回滚、三档审批安全等较复杂的子系统。

用户希望对这个项目做一次**全维度、全量代码、报告 + 立刻修复**的代码审查，并把修复直接 commit 到 `main`，最终报告只显示在聊天中（不落盘）。

### 1.1 范围
- **审查范围**：`src/` 全部代码（主进程 + 渲染端 + shared + agent 工具）
- **审查维度**（6 个）：安全 / 性能 / 架构 / 代码质量 / 测试 / 可维护性
- **执行方式**：6 个维度并行扫描 + 主协调者整合 + 修复落地
- **产出**：聊天中的最终报告 + `main` 分支上的修复 commit

### 1.2 不在范围
- 外部依赖升级（node-pty、Electron、SDK 等）
- 全新功能开发
- 大型架构重构（仅在修复必要依赖时做最小改动）
- 用户层面功能变更（不改动产品行为，只修 bug / 提质量）

## 2. 执行架构

### 2.1 总体流程

```
阶段 1 — 并行扫描
  6 个 Explore 子代理并发执行，每个负责一个维度的全量扫描
                ↓
阶段 2 — 整合去重
  主协调者（我）收集 6 份报告，去重、关联、分级、排序
                ↓
阶段 3 — 修复落地
  按 P0 → P1 → P2 顺序，每批修复后跑 tsc + vitest 验证
                ↓
阶段 4 — 验证 + 报告
  全量测试 + tsc，输出聊天报告（含已修 + 未修 + 维度小结 + 后续建议）
```

### 2.2 派遣技能
使用 `dispatching-parallel-agents` skill 来规范派遣 6 个 Explore 子代理，确保：
- 任务描述自包含
- 独立任务、无共享状态
- 返回结构化结果

## 3. 6 个并行代理的职责边界

每个代理的输出格式统一（去重的前置条件）：

```
[严重程度 P0/P1/P2] 标题
文件:行号 | 模块
问题描述（具体代码引用）
风险/影响
修复建议（具体改动方向）
```

| 代理 | 关注 | 不关注 |
|---|---|---|
| **安全** | IPC 边界、命令注入、路径穿越、symlink、密钥存储、XSS、preload 白名单 | 性能、命名风格 |
| **性能** | 大文件/大列表渲染、embedding 索引、agent 循环、缓存、内存泄漏、死循环 | UI 命名、API 设计 |
| **架构** | 模块边界、数据流、循环依赖、单一职责、接口清晰度 | 单文件内代码风格 |
| **代码质量** | 类型安全、错误处理、命名、可读性、重复代码、注释、ESLint 警告 | 整体架构、严重 bug |
| **测试覆盖** | 哪些模块有/无测试、关键路径覆盖、edge case、可测性 | 性能优化 |
| **可维护性** | 配置、依赖、文档、可观测性（log/error）、未来兼容性 | 短期 bug |

## 4. 主协调者（我）的责任

1. 用 `dispatching-parallel-agents` skill 派遣 6 个代理
2. 回收报告，做整合：
   - 去重（同问题被多个代理发现）
   - 关联（同 fix 解多问题）
   - 分级（严重程度 P0/P1/P2）
   - 排序（先修哪些后修哪些）
3. 执行修复（实际改代码 + 跑测试 + commit 到 main）
4. 输出最终报告（在聊天中）

## 5. 优先级定义

| 级别 | 含义 | 例子 | 修复时机 |
|---|---|---|---|
| **P0** | 安全漏洞、数据丢失风险、必崩 | 路径穿越、命令注入、内存泄漏未释放、未捕获 promise rejection | 第一批立刻修 |
| **P1** | 性能严重退化、明显 bug、坏味道 | N+1 查询、agent 死循环、竞态条件、大文件卡顿、UI 闪烁 | 第二批 |
| **P2** | 代码质量、可维护性、风格 | 重复代码、命名混乱、类型不严、文档缺失 | 第三批 |
| **P3** | 超出本次范围（标记但**不修**） | 依赖升级、大重构、跨子系统设计变更 | 不修，列入"推荐后续行动" |

## 6. 修复节奏

```
P0 批 1 ─── tsc + vitest ───→ commit
   ↓
P0 批 2 (如有更多) ─── tsc + vitest ───→ commit
   ↓
P1 批 1 ─── tsc + vitest ───→ commit
   ↓
P1 批 2 ... 继续直到 P1 全部修完或评估收尾
   ↓
P2 批 ... 可选，时间允许就修
   ↓
最终全量测试 + tsc 验证 → 输出报告
```

- 每批**修一组**相关的 P0（比如都是 IPC 问题）一起改，单个 commit
- 每批之间**必须**通过编译 + 测试，否则停下来定位，不强行 commit
- 修不动的（比如需要重构大块）会**降级为 P3 / 待办**，不进 commit

## 7. Commit 信息格式

```
fix(review): #<编号> <维度> — <一句话问题描述>

- 修复内容
- 关联 issue / 风险
- 验证：tsc 通过 / vitest 通过

Refs: code-review 2026-06-15
```

例：`fix(review): #003 security — preload 暴露的 IPC 缺少 path 校验`

## 8. 最终报告结构（聊天中输出）

```
## 代码审查报告 — 2026-06-15

### 概述
- 审查范围（src/、6 维度、6 代理并行）
- 6 维度发现统计（P0/P1/P2 数量 + 修复数）

### 已修复（X 个）
| # | 严重 | 维度 | 文件:行 | 标题 | commit |
|---|------|------|---------|------|--------|
| 1 | P0 | 安全 | preload.ts:42 | ... | abc1234 |

### 已识别但未修（X 个）
| # | 严重 | 维度 | 位置 | 原因（太复杂 / 风险高 / 偏离范围） |
|---|------|------|------|--------------------------------------|

### 维度小结
- 安全 / 性能 / 架构 / 代码质量 / 测试 / 可维护性 — 每个维度 2~3 段总结
- 整体趋势：哪些维度强、哪些弱

### 推荐后续行动
- 不在本次范围内的架构性建议
- 长期改进方向
```

## 9. 边界与异常处理

| 情况 | 处理 |
|---|---|
| 某个代理没发现该维度的问题 | 在报告"维度小结"中明确写"无显著问题" |
| P0 修复导致测试失败 | 停下来分析，不强行 commit；纳入"未修"并说明 |
| 单个修复需跨多个文件大改 | 评估工作量：若**该修复**手动改超过 ~15 分钟则降级为 P3 |
| 涉及外部依赖（如 node-pty 升级）| 不在本次范围，标 P3 建议 |
| 用户中途叫停 | 输出当前进度 + 已 commit 列表 |
